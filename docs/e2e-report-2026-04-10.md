## Failures

### FAIL-1: GET /api/reports/revenue returns 404
- **Severity**: Low
- **Category**: CRM API Endpoints
- **Expected**: 200 with revenue report data
- **Actual**: 404 Not Found
- **Root cause**: Route not implemented — spec/code gap
- **Impact**: None on live users (page not linked from UI)

### FAIL-2: POST /api/simulate/app-exception returns 200 (not 500)
- **Severity**: None (by design)
- **Category**: Simulation Lab — Chaos
- **Expected**: 500 (simulated server error)
- **Actual**: 200 with `{"status":"completed","type":"app_exception","error":"Simulated application exception..."}`
- **Root cause**: The endpoint generates APM error telemetry (records exception on the OTel span) but returns a success response. This is intentional — the goal is APM trace visibility, not an actual HTTP error.
- **Impact**: None

### FAIL-3: Shop pod sync_engine ORA-01017
- **Severity**: Medium
- **Category**: Database Health
- **Expected**: Sync engine can query Oracle ATP
- **Actual**: ORA-01017: invalid username/password; logon denied
- **Root cause**: Same KB-435 pattern as the CRM — the Shop's sync_engine has stale/incorrect credentials. The Shop's async engine and HTTP APIs work fine (all Category 2 tests pass) because the async connection pool has pre-existing connections.
- **Impact**: Low currently (Shop HTTP APIs functional), but will crash on next pod restart.
- **Remediation**: Update `drone-shop-secrets` in `octo-drone-shop` namespace with correct `ORACLE_PASSWORD`, then restart the deployment.

### FAIL-4: X-Correlation-Id not echoed back
- **Severity**: Low
- **Category**: Trace Correlation
- **Expected**: Server echoes client-supplied `X-Correlation-Id` in response
- **Actual**: Server generates its own `x-correlation-id` from the OTel trace context
- **Root cause**: The correlation middleware generates a new ID from the current trace span rather than preserving the client's ID. This is technically correct (server trace ID takes precedence) but prevents clients from correlating their own IDs with server traces.
- **Impact**: Minor — affects external monitoring tools that inject their own correlation IDs.

## Recommendations

1. **Shop sync_engine credentials** (FAIL-3): Patch `drone-shop-secrets` with current Oracle password and restart. Same fix as CRM KB-435.
2. **Revenue report route** (FAIL-1): Implement `/api/reports/revenue` or remove it from any documentation referencing it.
3. **Correlation ID echo** (FAIL-4): Consider preserving client-supplied `X-Correlation-Id` when present, falling back to the OTel trace ID when not. This improves interoperability with external APM tools.
